실시간 음성 입력을 붙일 때 가장 먼저 막히는 부분은 모델보다 전송 구조입니다. 녹음이 끝난 뒤 파일 전체를 업로드하면 구현은 쉽지만, 자막이나 음성 명령이 몇 초씩 늦어집니다. Vercel은 2026년 7월 22일 AI Gateway에 스트리밍 전사를 추가했고, AI SDK의 `streamTranscribe`를 사용하면 **오디오를 캡처하는 동안 전송하고 전사 델타를 즉시 받을 수 있게 됐습니다.**

다만 이 기능은 아직 beta이며 API 이름도 `experimental_streamTranscribe`입니다. 운영 환경에서는 세 가지를 먼저 고정해야 합니다.

1. 브라우저나 장치가 보내는 오디오 형식
2. 장기 API 키를 클라이언트에 노출하지 않는 인증 구조
3. 부분 전사와 최종 전사를 분리하는 상태 관리

이 글에서는 공식 예제를 그대로 복사하는 데서 끝내지 않고, 실제 음성 입력 기능으로 연결할 때 필요한 경계를 정리합니다.

## 기존 파일 전사와 무엇이 달라졌나

기존 `transcribe()` 방식은 완성된 오디오 파일을 입력하고 전체 결과를 한 번에 받습니다. 회의 녹음이나 업로드된 음성 메모에는 적합하지만, 사용자가 말하는 동안 화면에 자막을 보여주거나 에이전트 명령을 미리 준비하기는 어렵습니다.

스트리밍 전사는 입력과 출력이 모두 흐름으로 동작합니다.

| 구분 | 파일 전사 | 스트리밍 전사 |
|---|---|---|
| 입력 | 완성된 파일·Buffer·URL | `ReadableStream` 오디오 청크 |
| 출력 | 완료된 전체 텍스트 | 전사 델타와 최종 텍스트 |
| 적합한 용도 | 업로드 파일, 회의록 후처리 | 실시간 자막, 음성 검색, 에이전트 음성 입력 |
| 구현 난이도 | 낮음 | 오디오 포맷·연결·상태 처리가 필요 |
| 현재 상태 | 모델별 지원 | beta, streaming-capable 모델만 지원 |

중요한 점은 스트리밍 전사가 곧바로 양방향 음성 에이전트를 의미하지는 않는다는 것입니다. `streamTranscribe`는 음성을 텍스트로 바꿉니다. 에이전트는 여전히 텍스트를 입력받고, 음성으로 답해야 한다면 별도의 speech generation을 연결해야 합니다. 지연 시간이 매우 짧은 완전한 양방향 대화가 목표라면 realtime voice 모델이 더 적합할 수 있습니다.

## 최소 실행 예제

Vercel의 출시 예제는 24kHz raw PCM 스트림을 `openai/gpt-realtime-whisper`에 보내고 `transcript-delta` 이벤트를 읽습니다.

```ts
import { experimental_streamTranscribe as streamTranscribe } from "ai";

export async function transcribePcm(
  audioStream: ReadableStream<Uint8Array>,
) {
  const result = streamTranscribe({
    model: "openai/gpt-realtime-whisper",
    audio: audioStream,
    inputAudioFormat: {
      type: "audio/pcm",
      rate: 24_000,
    },
  });

  let partialText = "";

  for await (const part of result.fullStream) {
    if (part.type === "transcript-delta") {
      partialText += part.delta;
      process.stdout.write(part.delta);
    }
  }

  const finalText = await result.text;

  return {
    partialText,
    finalText,
  };
}
```

`partialText`는 사용자에게 빠르게 보여주기 위한 값입니다. 저장, 검색, 도구 실행처럼 되돌리기 어려운 작업에는 `finalText`를 사용해야 합니다. 부분 전사는 뒤 문맥에 따라 수정될 수 있기 때문입니다.

## 클라이언트에 AI Gateway 키를 넣지 않는다

브라우저에서 WebSocket 전사를 직접 시작하려고 장기 AI Gateway 키를 전달하면 안 됩니다. 공식 Grok STT 예제는 서버에서 짧은 수명의 토큰을 발급한 뒤 클라이언트용 Gateway 인스턴스를 만드는 방식을 사용합니다.

```ts
// server/transcription-token.ts
import { gateway } from "@ai-sdk/gateway";

const modelId = "xai/grok-stt";

export async function createTranscriptionToken() {
  const { token } = await gateway.experimental_transcription.getToken({
    model: modelId,
  });

  return {
    modelId,
    token,
  };
}
```

클라이언트나 제한된 실행 환경에서는 전달받은 토큰만 사용합니다.

```ts
import { experimental_streamTranscribe as streamTranscribe } from "ai";
import { createGateway } from "@ai-sdk/gateway";

export function startTranscription(
  modelId: string,
  token: string,
  audio: ReadableStream<Uint8Array>,
) {
  const clientGateway = createGateway({ apiKey: token });

  return streamTranscribe({
    model: clientGateway.transcription(modelId),
    audio,
    inputAudioFormat: {
      type: "audio/pcm",
      rate: 24_000,
    },
  });
}
```

이 구조의 보안 경계는 명확합니다.

- 장기 Gateway 자격 증명은 서버에만 보관합니다.
- 토큰 발급 API에는 사용자 인증과 rate limit을 적용합니다.
- 토큰은 요청한 모델과 기능에 필요한 범위로 제한합니다.
- 오디오 원본 보관 여부와 보존 기간을 제품 정책으로 명시합니다.

## PCM 형식을 먼저 맞춰야 한다

공식 예제의 입력은 **24kHz, 16-bit signed little-endian, mono raw PCM**입니다. 파일 확장자만 `.pcm`으로 바꾸거나 압축된 음성을 그대로 흘려보내면 정상 전사를 기대할 수 없습니다.

```text
샘플 레이트: 24,000Hz
채널: mono
샘플: signed 16-bit little-endian
컨테이너: 없음(raw PCM)
```

브라우저 녹음 API나 모바일 SDK가 WebM, Opus, AAC 같은 압축 형식을 반환한다면 다음 중 하나를 선택해야 합니다.

- 캡처 단계에서 모델이 지원하는 PCM으로 변환
- 서버에서 디코딩·리샘플링 후 Gateway로 전달
- 해당 입력 형식을 직접 지원하는 스트리밍 전사 모델 선택

형식 변환은 CPU 사용량과 지연을 늘립니다. 제품을 만들기 전에 대상 브라우저·장치에서 실제 출력 형식을 수집하고, 전사 모델이 요구하는 형식과 비교하는 편이 안전합니다.

## 부분 자막과 최종 자막을 분리한다

UI에서 한 개의 문자열만 계속 이어 붙이면 중복 문장이나 확정되지 않은 단어가 저장될 수 있습니다. 상태를 최소 두 층으로 나누는 편이 좋습니다.

```ts
type TranscriptState = {
  interim: string;
  final: string;
};

function appendDelta(
  state: TranscriptState,
  delta: string,
): TranscriptState {
  return {
    ...state,
    interim: state.interim + delta,
  };
}

function finalizeTranscript(
  state: TranscriptState,
  text: string,
): TranscriptState {
  return {
    interim: "",
    final: text,
  };
}
```

권장 사용 방식은 다음과 같습니다.

- `interim`: 화면 자막, 입력 중 표시, 검색어 미리보기
- `final`: 메시지 전송, 데이터베이스 저장, 에이전트 실행
- 원본 오디오: 필요한 경우에만 별도 동의 후 저장

도구 호출 에이전트라면 부분 전사만으로 구매, 삭제, 배포 같은 작업을 실행하지 않는 것이 좋습니다. 최종 전사가 도착한 뒤 명령을 다시 분류하고, 쓰기 작업에는 승인 단계를 둡니다.

## 음성 입력을 기존 텍스트 에이전트에 연결하기

스트리밍 전사의 장점은 에이전트 구조를 바꾸지 않아도 된다는 점입니다. 음성을 텍스트로 정규화한 뒤 기존 에이전트 입력 경로로 전달합니다.

```ts
async function runVoiceCommand(
  audio: ReadableStream<Uint8Array>,
) {
  const { finalText } = await transcribePcm(audio);

  if (!finalText.trim()) {
    return {
      status: "empty" as const,
    };
  }

  // 기존 텍스트 에이전트 호출부를 그대로 재사용합니다.
  const response = await runTextAgent({
    message: finalText,
    source: "voice",
  });

  return {
    status: "completed" as const,
    transcript: finalText,
    response,
  };
}
```

프로덕션에서는 다음 정보를 함께 기록하면 장애를 찾기 쉽습니다.

- 모델 ID와 실제 provider
- 입력 오디오 형식과 샘플 레이트
- 첫 전사 델타까지 걸린 시간
- 최종 전사 완료 시간
- 연결 종료·재시도 사유
- 빈 전사 비율
- 사용자가 수정한 최종 문장 비율

정확도만 측정하면 실시간 제품의 문제를 놓칠 수 있습니다. 첫 델타가 늦거나 연결이 자주 끊기면 최종 문장이 정확해도 사용 경험은 좋지 않습니다.

## 모델을 바꿀 때 확인할 것

같은 `streamTranscribe` 코드를 사용해도 모든 전사 모델의 동작이 같지는 않습니다.

1. 스트리밍 지원 여부
2. 입력 오디오 포맷과 샘플 레이트
3. 언어 지원 범위
4. 부분 결과의 안정성
5. 비용 계산 기준
6. 지역·보존·학습 정책
7. 최대 세션 길이와 연결 제한

Vercel은 모델 문자열을 바꾸면 `xai/grok-stt` 등 다른 streaming-capable 모델을 사용할 수 있다고 설명합니다. 그러나 모델 교체는 텍스트 모델 교체보다 입력 형식과 세션 동작의 영향을 많이 받습니다. 최소한 실제 마이크 환경, 잡음 환경, 짧은 명령, 긴 발화로 나눠 평가해야 합니다.

## 운영 체크리스트

- [ ] `experimental_streamTranscribe`가 beta라는 점을 반영해 버전을 고정했다.
- [ ] 장기 API 키를 클라이언트 번들에 포함하지 않았다.
- [ ] 오디오 형식과 샘플 레이트를 모델 요구 사항과 맞췄다.
- [ ] 부분 전사와 최종 전사를 별도 상태로 관리한다.
- [ ] 연결 종료·재접속·빈 전사 처리가 있다.
- [ ] 쓰기 도구는 최종 전사와 사용자 승인 뒤 실행한다.
- [ ] 오디오·전사 데이터의 보존 및 삭제 정책을 정했다.
- [ ] 첫 델타 지연과 최종 완료 시간을 함께 관측한다.
- [ ] 모델 변경 시 실제 음성 데이터로 회귀 테스트한다.

## 결론

AI Gateway의 스트리밍 전사는 기존 텍스트 에이전트에 음성 입력을 붙이는 경로를 단순하게 만듭니다. 하지만 핵심은 함수 한 번을 호출하는 것이 아니라 **오디오 형식, 임시 인증, 부분·최종 상태의 경계를 명확히 두는 것**입니다.

먼저 24kHz PCM으로 작은 서버 측 예제를 통과시키고, 다음으로 짧은 수명의 클라이언트 토큰과 UI 상태를 붙이는 순서가 안전합니다. 완전한 양방향 음성 대화가 필요할 때만 realtime voice 구조로 확장하는 편이 좋습니다.

AI Gateway에서 지연 시간과 비용을 분리하는 방법은 [Service Tiers 운영 가이드](/blog/vercel-ai-gateway-service-tiers-guide/)에서, 에이전트 승인과 durable 실행 구조는 [AI SDK 7 프로덕션 에이전트 가이드](/blog/ai-sdk-7-production-agent-guide/)에서 이어서 볼 수 있습니다.
