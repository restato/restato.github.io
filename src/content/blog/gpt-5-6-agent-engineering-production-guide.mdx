GPT-5.6의 가장 큰 숫자는 105만 토큰 컨텍스트입니다. 하지만 운영 환경에서 더 중요한 변화는 따로 있습니다. 웹과 파일을 검색하고, 코드를 실행하고, 패치를 적용하며, 컴퓨터와 MCP 도구까지 사용할 수 있다는 점입니다.

이 정도면 모델을 단순한 답변 API로 취급하기 어렵습니다. **GPT-5.6 에이전트의 품질은 프롬프트보다 모델 라우팅, 도구 권한, 검증 루프에서 더 크게 갈립니다.**

Sol, Terra, Luna의 가격과 선택 기준이 궁금하다면 [GPT-5.6 모델 선택 가이드](/blog/gpt-5-6-sol-terra-luna-api-guide/)를 먼저 참고할 수 있습니다. 이 글은 그다음 단계인 운영 구조에 집중합니다.

## GPT-5.6을 답변 API로만 쓰면 놓치는 것

GPT-5.6 제품군은 텍스트와 이미지 입력, 최대 105만 토큰 컨텍스트, 최대 12만 8천 토큰 출력을 지원합니다. Sol 모델은 공식 문서에서 복잡한 전문 업무를 위한 프런티어 모델로 소개되며, `gpt-5.6` 별칭도 Sol을 가리킵니다.

Responses API에서 사용할 수 있는 도구의 범위는 다음과 같습니다.

- 웹 검색과 파일 검색
- 코드 인터프리터와 호스팅 셸
- 이미지 생성
- 패치 적용
- 컴퓨터 사용
- MCP와 도구 검색

예전의 LLM 애플리케이션은 사용자가 질문하면 모델이 텍스트를 반환하는 구조가 많았습니다. 에이전트는 다릅니다. 모델이 중간 결과를 보고 다음 행동을 고르며, 외부 시스템의 상태를 실제로 바꿀 수 있습니다.

```text
질문 → 모델 → 답변
```

이 구조가 다음처럼 바뀝니다.

```text
요청
  ↓
계획 수립
  ↓
도구 선택 → 실행 → 결과 관찰
  ↑                  ↓
  └──── 계획 수정 ───┘
  ↓
검증
  ↓
최종 결과
```

모델의 능력이 좋아져도 실행 경계가 허술하면 시스템은 불안정합니다. 반대로 권한과 검증을 잘 설계하면 Terra나 Luna도 큰 시스템의 일부로 충분히 활용할 수 있습니다.

## 모델 선택은 요청 단위로 해야 한다

모든 요청을 Sol에 보내는 방식은 구현하기 쉽지만 비용이 큽니다. 모든 요청을 Luna로 처리하면 복잡한 작업에서 실패율이 올라갈 수 있습니다. 세 모델을 역할별로 배치하는 편이 낫습니다.

| 작업 특성 | 시작 모델 | 예시 |
|---|---|---|
| 실패 비용이 크고 여러 단계를 계획해야 함 | Sol | 코드베이스 수정, 보안 분석, 복잡한 조사 |
| 일반적인 제품 기능과 업무 자동화 | Terra | 문서 분석, 검색 기반 답변, 코드 리뷰 |
| 대량의 짧고 정형화된 처리 | Luna | 분류, 추출, 라우팅, 형식 변환 |

라우팅 기준은 입력 길이만으로 정하지 않습니다. 다음 값을 함께 봐야 합니다.

- 요청의 위험 등급
- 필요한 도구와 권한
- 결과를 자동으로 검증할 수 있는지
- 실패했을 때 재시도가 가능한지
- 사용자 응답 시간 제한
- 요청당 허용 비용

간단한 라우터는 다음처럼 만들 수 있습니다.

```ts
type Risk = "low" | "medium" | "high";
type Workload = "classify" | "answer" | "execute";

function selectModel(risk: Risk, workload: Workload): string {
  if (risk === "high" || workload === "execute") {
    return "gpt-5.6-sol";
  }

  if (workload === "classify") {
    return "gpt-5.6-luna";
  }

  return "gpt-5.6-terra";
}
```

이 코드는 출발점일 뿐입니다. 실제 서비스에서는 평가 결과를 바탕으로 라우팅 규칙을 조정해야 합니다. 예를 들어 Terra의 코드 리뷰가 특정 저장소에서 반복적으로 실패할 때만 Sol로 승격할 수 있습니다.

## 도구는 최소 권한으로 연결한다

웹 검색은 정보를 읽지만, 셸과 컴퓨터 사용 도구는 환경을 변경할 수 있습니다. 같은 도구 목록을 모든 요청에 제공하면 프롬프트 인젝션이나 잘못된 판단이 실제 사고로 이어질 수 있습니다.

도구를 세 등급으로 나누면 경계를 세우기 쉽습니다.

| 등급 | 권한 | 예시 |
|---|---|---|
| 읽기 | 외부 상태를 바꾸지 않음 | 웹 검색, 파일 검색, 로그 조회 |
| 제한된 쓰기 | 격리된 범위에서만 변경 | 임시 파일 생성, 테스트 저장소 패치 |
| 승인 필요 | 운영 환경이나 외부 시스템 변경 | 배포, 데이터 삭제, 결제, 메시지 전송 |

에이전트가 코드 수정 작업을 수행한다면 다음 조건을 권장합니다.

1. 저장소마다 별도 작업 디렉터리나 컨테이너를 사용합니다.
2. 비밀 값은 모델 컨텍스트와 셸 환경에서 제거합니다.
3. 수정 가능한 경로를 허용 목록으로 제한합니다.
4. 네트워크 접근은 필요한 도메인만 허용합니다.
5. 운영 배포와 데이터 삭제에는 사람의 승인을 둡니다.
6. 실행한 도구와 인자를 감사 로그에 남깁니다.

도구 설명도 권한의 일부입니다. "필요하면 셸을 사용하라"보다 "테스트와 정적 분석만 실행할 수 있으며 배포 명령은 금지한다"처럼 범위를 구체적으로 적어야 합니다.

## 계획, 실행, 검증을 분리한다

한 번의 모델 호출에 계획과 실행, 자체 검증을 모두 맡기면 실패 원인을 찾기 어렵습니다. 운영 에이전트는 세 단계를 분리하는 편이 좋습니다.

### 1. 계획 단계

계획 단계는 무엇을 변경할지 정하지만 실제 도구는 실행하지 않습니다.

출력에는 최소한 다음 항목이 있어야 합니다.

- 목표와 완료 조건
- 조사할 파일이나 데이터
- 사용할 도구
- 예상되는 부작용
- 검증 방법

### 2. 실행 단계

실행기는 계획을 입력받아 허용된 도구만 호출합니다. 모델이 새로운 작업을 발견해도 권한 범위를 넓히지 않고 계획 단계로 되돌려야 합니다.

웹 검색을 허용한 간단한 Responses API 호출은 다음과 같습니다.

```ts
import OpenAI from "openai";

const client = new OpenAI();

const response = await client.responses.create({
  model: "gpt-5.6-terra",
  reasoning: { effort: "medium" },
  tools: [{ type: "web_search" }],
  input: `공식 문서만 사용해 이 라이브러리의 최신 마이그레이션 변경점을 조사하고,
출처 URL과 함께 위험 항목을 정리해 주세요.`,
});

console.log(response.output_text);
```

도구 목록을 요청별로 동적으로 구성하는 것이 중요합니다. 단순 분류 요청에 셸이나 컴퓨터 사용 권한을 줄 이유는 없습니다.

### 3. 검증 단계

검증기는 모델의 설명이 아니라 외부 신호를 확인합니다.

- 코드 변경은 테스트와 타입 검사로 검증합니다.
- 구조화 데이터는 JSON Schema로 검증합니다.
- 문서 요약은 원문 인용 위치를 대조합니다.
- 데이터 변경은 변경 전후 레코드 수를 비교합니다.
- 웹 조사는 링크와 발행일을 다시 확인합니다.

검증 실패 시 같은 요청을 그대로 반복하지 않습니다. 실패 로그를 다음 시도에 넣고, 재시도 횟수와 비용 상한을 제한해야 합니다.

```text
실행 결과
  ├─ 검증 성공 → 결과 반환
  └─ 검증 실패
       ├─ 재시도 가능 → 실패 근거를 포함해 재계획
       └─ 재시도 한도 초과 → 사람에게 전달
```

## 105만 토큰을 전부 채우지 말 것

긴 컨텍스트는 대규모 저장소와 여러 문서를 한 번에 분석할 때 유용합니다. 그렇다고 검색과 요약이 필요 없어지는 것은 아닙니다.

GPT-5.6은 입력이 27만 2천 토큰을 넘으면 해당 요청 전체에 입력 2배, 출력 1.5배 요율이 적용됩니다. 관련성이 낮은 문서까지 한꺼번에 넣으면 정확도, 지연 시간, 비용이 함께 나빠질 수 있습니다.

긴 컨텍스트는 다음 순서로 사용하는 편이 안전합니다.

1. 파일 검색이나 인덱스로 후보 문서를 좁힙니다.
2. 파일 경로, 날짜, 작성자 같은 메타데이터를 유지합니다.
3. 고정 지침과 반복 문서는 캐시 가능한 앞부분에 둡니다.
4. 사용자 입력과 도구 결과처럼 매번 바뀌는 내용은 뒤에 둡니다.
5. 전체 문맥이 필요한 작업만 27만 2천 토큰 초과 비용을 허용합니다.

캐시도 무조건 이득은 아닙니다. GPT-5.6 Sol의 캐시 읽기 가격은 일반 입력의 10분의 1이지만, 첫 캐시 쓰기는 일반 입력의 1.25배입니다. 한 번 읽고 버릴 문서는 캐시하지 않는 편이 쌀 수 있습니다.

## 에이전트 평가는 정답률 하나로 부족하다

에이전트는 여러 번 도구를 호출하기 때문에 최종 답변만 평가하면 중간 과정의 문제를 놓칩니다. 최소한 다음 지표를 함께 기록해야 합니다.

| 지표 | 확인할 질문 |
|---|---|
| 작업 성공률 | 완료 조건을 실제로 충족했는가 |
| 도구 성공률 | 올바른 도구와 인자를 선택했는가 |
| 검증 통과율 | 테스트나 스키마 검증을 통과했는가 |
| 불필요한 행동 수 | 필요 없는 검색과 파일 수정을 했는가 |
| 승인 요청률 | 사람이 개입해야 한 작업이 얼마나 되는가 |
| 요청당 성공 비용 | 실패와 재시도를 포함해 얼마가 들었는가 |
| 지연 시간 | 평균과 상위 95% 완료 시간은 얼마인가 |

코딩 에이전트라면 테스트 통과 여부와 변경 파일 수를 함께 봐야 합니다. 테스트만 통과하면서 요구하지 않은 리팩터링을 많이 하는 모델은 운영하기 어렵습니다.

문서 조사 에이전트라면 인용 정확도와 출처 누락률이 중요합니다. 답변이 자연스러운지보다 각 주장을 원문에서 확인할 수 있는지가 먼저입니다.

평가 데이터는 실제 운영 요청에서 만들되 개인정보와 비밀 값을 제거해야 합니다. Sol, Terra, Luna를 같은 데이터로 실행하고 품질과 비용을 함께 비교하면 라우팅 기준을 구체화할 수 있습니다.

## 운영 체크리스트

GPT-5.6 에이전트를 배포하기 전에 다음 항목을 확인합니다.

- [ ] 요청별로 모델과 추론 강도를 선택한다.
- [ ] 도구 권한을 읽기, 제한된 쓰기, 승인 필요로 구분한다.
- [ ] 셸과 컴퓨터 사용은 격리된 환경에서 실행한다.
- [ ] 비밀 값이 프롬프트와 도구 결과에 포함되지 않는다.
- [ ] 최대 도구 호출 수와 재시도 횟수를 제한한다.
- [ ] 코드, JSON, 인용을 외부 검증기로 확인한다.
- [ ] 27만 2천 토큰 초과 요율을 비용 계산에 반영한다.
- [ ] 캐시 쓰기와 도구 호출 비용을 별도로 기록한다.
- [ ] 모델 별칭 변경에 대비해 회귀 평가를 자동화한다.
- [ ] 실패 시 사람이 이어받을 수 있는 로그와 상태를 남긴다.

## 결론

GPT-5.6의 긴 컨텍스트와 넓어진 도구 지원은 에이전트가 처리할 수 있는 작업 범위를 키웠습니다. 동시에 잘못된 행동이 미치는 범위도 커졌습니다.

실용적인 시작점은 Terra에 읽기 도구만 연결하고, 결과를 자동 검증하는 작은 루프를 만드는 것입니다. 대량의 정형 작업은 Luna로 분리하고, 평가에서 실제 차이가 확인된 고난도 작업만 Sol로 올립니다. 쓰기 권한과 운영 시스템 변경은 마지막에 추가합니다.

에이전트의 신뢰성은 모델 이름 하나로 결정되지 않습니다. 모델이 무엇을 볼 수 있는지, 무엇을 바꿀 수 있는지, 실패를 어떻게 발견하는지가 운영 품질을 결정합니다.
