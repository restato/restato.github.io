# Restato Blog Writer

## 목적

`restato/restato.github.io`에 검색 가치가 있고 실제 경험이 느껴지는 한국어 개발 글을 꾸준히 발행한다. 단순한 AI 요약문이나 키워드 채우기 글은 작성하지 않는다.

## 적용 범위

다음 작업을 할 때 이 스킬을 반드시 먼저 읽고 따른다.

- 새 블로그 주제 선정
- `src/content/blog/*.mdx` 글 작성 또는 수정
- 자동 포스팅
- 개발 프로젝트 회고 및 튜토리얼 작성
- AI, Claude, Codex, Vibe Coding 활용기 작성

## 1. 작업 전 조사

1. `src/content/blog/`의 기존 글 제목, 설명, 태그와 최근 글을 확인한다.
2. 같은 검색 의도나 결론을 가진 글이 있으면 새 글을 만들지 않는다.
3. 저장소의 최근 커밋과 실제 구현 내용을 확인해 글의 근거로 활용한다.
4. 시의성이 있는 제품, API, 버전, 가격, 정책은 공식 문서에서 최신 정보를 확인한다.
5. 직접 확인하지 못한 결과나 사용 경험을 실제 경험처럼 쓰지 않는다.

## 2. 주제 선정 원칙

우선순위는 다음과 같다.

1. 이 저장소에서 실제로 구현하거나 해결한 문제
2. 반복 작업을 자동화한 과정과 시행착오
3. Claude, Codex 등 AI 도구를 개발에 적용한 구체적인 방법
4. Astro, React, TypeScript, GitHub Pages 운영 노하우
5. 소규모 사업, 판매, 업무 자동화에 개발을 적용한 사례

좋은 주제는 다음 질문에 명확히 답해야 한다.

- 독자가 어떤 문제를 해결할 수 있는가?
- Restato가 직접 해본 내용은 무엇인가?
- 다른 글과 다른 구체적인 포인트는 무엇인가?
- 글을 읽고 바로 실행할 수 있는가?

다음 유형은 피한다.

- 근거 없는 미래 예측
- 공식 문서를 다시 요약한 글
- 내용이 거의 같은 시리즈 글의 남발
- 제목만 자극적인 글
- 확인하지 않은 성능 비교
- 의미 없는 “Top 10” 나열

## 3. 문체

- 한국어로 작성한다.
- 1인칭 경험은 실제 저장소나 작업 기록으로 확인될 때만 사용한다.
- 친근하지만 과장하지 않는다.
- 첫 문단에서 문제와 글의 결론을 바로 제시한다.
- 긴 배경 설명보다 실행 과정, 코드, 판단 근거를 우선한다.
- 문단은 짧게 유지하고 소제목으로 흐름을 나눈다.
- “혁신적”, “완벽한”, “놀라운” 같은 홍보성 표현은 피한다.
- AI가 작성했다는 상투적인 문구를 넣지 않는다.

## 4. 권장 글 구조

글마다 필요한 부분만 사용하되 대체로 다음 순서를 따른다.

1. 해결하려던 문제
2. 최종 결과 또는 핵심 결론
3. 선택한 접근법과 이유
4. 실제 구현 단계
5. 코드 또는 설정 예시
6. 막혔던 점과 해결 방법
7. 운영 시 주의점
8. 다음 개선 계획

튜토리얼이라면 독자가 복사해서 실행할 수 있도록 전제 조건과 파일 경로를 명확히 쓴다.

## 5. MDX 형식

파일 위치:

`src/content/blog/<영문-kebab-case-slug>.mdx`

Frontmatter:

```mdx
---
title: "구체적인 결과가 드러나는 제목"
description: "독자가 얻게 될 결과를 80~150자 내외로 설명"
pubDate: 2026-07-18
tags: ["AI", "Astro", "Automation"]
---
```

규칙:

- 날짜 필드는 `pubDate`를 사용한다.
- 파일명은 영문 소문자 kebab-case로 작성한다.
- 제목은 기존 글과 중복되지 않게 한다.
- 태그는 2~5개로 제한하며 기존 태그를 우선 재사용한다.
- H1은 레이아웃에서 제공하므로 본문에서 `#` 제목을 반복하지 않는다.
- 코드 블록에는 가능한 한 언어를 지정한다.
- 존재하지 않는 컴포넌트나 import를 추가하지 않는다.
- MDX에서 깨질 수 있는 JSX 기호는 이스케이프하거나 코드 블록에 넣는다.

## 6. SEO 기준

- 제목에는 핵심 문제나 결과를 자연스럽게 포함한다.
- description은 제목을 반복하지 말고 독자의 이득을 설명한다.
- 첫 150자 안에 핵심 키워드와 결론이 드러나게 한다.
- 관련 기존 글이 있으면 자연스러운 내부 링크를 1~3개 넣는다.
- 외부 링크는 공식 문서와 1차 자료를 우선한다.
- 검색 키워드를 억지로 반복하지 않는다.
- 얇은 콘텐츠를 피하고, 최소한 하나 이상의 고유한 코드·절차·판단 근거를 포함한다.

## 7. 사실성과 코드 품질

- 저장소에 없는 기능을 이미 구현된 것처럼 쓰지 않는다.
- 코드 예시는 현재 프로젝트의 Astro 5, React 19, TypeScript 환경에 맞춘다.
- 명령어와 파일 경로는 실제 저장소 기준으로 확인한다.
- 외부 제품 정보는 공식 문서를 기준으로 작성한다.
- 수치, 성능, 비용을 언급할 때는 출처 또는 측정 조건을 명시한다.
- 비밀키, 토큰, 이메일, 개인 데이터는 절대 포함하지 않는다.

## 8. 발행 전 체크리스트

- [ ] 기존 글과 주제 및 검색 의도가 겹치지 않는다.
- [ ] `title`, `description`, `pubDate`, `tags`가 모두 있다.
- [ ] 파일명이 영문 kebab-case다.
- [ ] 첫 문단에 문제와 결론이 있다.
- [ ] 독자에게 새로운 코드, 절차 또는 판단 근거를 제공한다.
- [ ] 확인하지 않은 경험이나 결과가 없다.
- [ ] 코드와 경로가 현재 저장소 구조와 맞는다.
- [ ] MDX 문법이 유효하다.
- [ ] 가능하면 `npm run build`를 통과한다.
- [ ] 변경 파일과 커밋 메시지가 글 내용에 맞는다.

## 9. 배포 방식

1. 기본 대상 저장소는 `restato/restato.github.io`다.
2. 실제 기본 브랜치를 저장소 메타데이터로 다시 확인한다.
3. 새 글 한 편만 추가하는 경우 작은 단일 커밋으로 유지한다.
4. 커밋 메시지는 `content: add <slug> post` 형식을 권장한다.
5. 푸시 후 GitHub Pages 배포 상태를 확인할 수 있으면 확인한다.
6. 완료 보고에는 제목, 파일 경로, 커밋 SHA, 예상 게시 URL을 포함한다.

## 10. 자동 포스팅 추가 규칙

- 하루에 최대 1편만 게시한다.
- 적절한 주제가 없거나 근거가 부족하면 억지로 게시하지 않고 그 이유를 보고한다.
- 연속 3편 이상 같은 핵심 태그를 사용하지 않는다.
- 최근 글과 제목 패턴, 도입부, 결론이 반복되지 않게 한다.
- 자동 작성 후에도 발행 전 체크리스트를 모두 통과해야 한다.
